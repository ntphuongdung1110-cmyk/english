## 📚 PREP Method

```
P — Point     → Nêu quan điểm / kết luận trước
R — Reason    → Giải thích lý do
E — Example   → Ví dụ cụ thể / bằng chứng
P — Point     → Nhắc lại quan điểm (để nhấn mạnh)
```

### Ví Dụ PREP Cho QC Manager

**Câu hỏi:** "Why do we need more time for testing?"

```
P: "I recommend we extend testing by 2 days before release."

R: "The reason is we've found 3 critical bugs in the last 48 hours — 
    all in the payment flow — and we haven't finished regression on mobile."

E: "Last quarter, we released with a similar risk profile. 
    Within 24 hours of launch, we had 200 user complaints 
    and had to push a hotfix — costing us 3 days of dev time."

P: "So I strongly recommend a 2-day extension — 
    it's a much smaller cost than dealing with production issues."
```

## 📚 Từ Vựng Hôm Nay — Presentation Signposting

| Từ / Cụm | Phiên âm | Nghĩa |
|---|---|---|
| **I'd like to make one key point...** | /aɪd laɪk tuː meɪk wʌn kiː pɔɪnt/ | Mở với điểm chính |
| **The reason I say that is...** | /ðə ˈriːzən aɪ seɪ ðæt ɪz/ | Giải thích |
| **To give you a concrete example...** | /tuː ɡɪv juː ə kənˈkriːt ɪɡˈzɑːmpəl/ | Ví dụ |
| **So in summary...** | /soʊ ɪn ˈsʌməri/ | Tóm lại |
| **What this means for us is...** | /wɒt ðɪs miːnz fɔːr ʌs ɪz/ | Ý nghĩa thực tế |
| **The bottom line is...** | /ðə ˈbɒtəm laɪn ɪz/ | Kết luận cuối |
| **If I had to leave you with one thing...** | /ɪf aɪ hæd tuː liːv juː wɪð wʌn θɪŋ/ | Takeaway chính |

## 🗣️ Mẫu Câu — PREP in Action

### Về Chất Lượng Sản Phẩm
```
P: "Quality is not just QC's job — it's everyone's responsibility."
R: "Because bugs found later in the cycle cost 10x more to fix 
    than bugs caught during development."
E: "In sprint 12, a requirement gap caught by QC at UAT 
    led to 3 days of rework that could have been avoided 
    with clearer acceptance criteria upfront."
P: "So quality has to start from the requirements stage — 
    not end at the testing stage."
```

## 🎧 Shadowing Script — PREP Presentation

```
"I want to make one key recommendation today.

We should implement mandatory test case reviews 
before any sprint begins. That's my point.

The reason is simple: our biggest source of test quality issues 
isn't bad testing — it's unclear test objectives. 
When test cases aren't reviewed, testers make assumptions, 
and important scenarios get missed.

To give you a concrete example: 
in our last sprint, we missed 4 edge cases 
in the subscription flow because the test cases 
were written based on outdated requirements. 
All 4 became production bugs.

So — mandatory test case reviews before sprint start. 
It adds one day of effort. It saves us 2–3 days of fire-fighting.
That's my recommendation."
```
