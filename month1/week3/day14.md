## 📚 SBI Model — Cách Feedback Chuyên Nghiệp Nhất

```
S — Situation  (Tình huống cụ thể, không chung chung)
B — Behavior   (Hành vi quan sát được, không phán xét)
I — Impact     (Ảnh hưởng của hành vi đó)
```

**Sai:**
> "Your bug reports are always incomplete."

**Đúng (SBI):**
> "In yesterday's standup (S), I noticed the bug report you submitted didn't include reproduction steps (B). That made it harder for the dev team to prioritize and fix it quickly (I)."

## 🗣️ Mẫu Câu Hôm Nay

### Pattern 1: Positive Feedback (Cụ Thể)
```
"I wanted to call out [specific action] — that was really well done.
The reason it stood out is [impact/result]."
→ "I wanted to call out the way you documented that edge case — really well done.
    The reason it stood out is it helped the dev team fix the bug in half the time."
```

### Pattern 2: SBI Constructive Feedback
```
"In [situation], I noticed [behavior]. The impact was [effect].
Going forward, I'd suggest [specific improvement]."
→ "In last sprint's regression run, I noticed several test cases were marked 'Pass' 
    without screenshots. The impact was we couldn't verify if the issue was truly resolved.
    Going forward, could you include at least one screenshot for each test result?"
```

### Pattern 3: Balanced Feedback (Sandwich)
```
"I want to start with what I think you're doing really well — [positive].
One thing I'd like us to work on is [area].
I'm confident you can get there because [encouragement]."
```

### Pattern 4: Feedback → Offer Support
```
"I've noticed [observation]. I wanted to bring it up because [reason].
Is there anything making that difficult for you right now?
How can I help?"
```

## 🎧 Shadowing Script — Feedback Conversation

```
"Hey, do you have a few minutes? 
I wanted to share some feedback from this week — 
nothing urgent, just wanted to check in.

First — I really appreciate how thoroughly 
you went through the payment flow scenarios. 
That level of detail is exactly what we need.

That said, I did want to flag one thing. 
In a couple of the reports, the reproduction steps 
weren't included. I get it — it's time-consuming — 
but without them, the devs had to loop back to us 
to understand the bug, which slows everyone down.

Going forward, even a short numbered list works. 
Something like: step 1, step 2, expected result, 
actual result. Does that sound manageable?

Is there anything on your end that's making it 
harder to get those in? I want to make sure 
you have what you need to do this well."
```

## 💬 Speaking Prompt — Roleplay (Ghi Âm 3 Phút)

> **Tình huống:** Một junior tester trong team bạn hay submit bug reports thiếu thông tin, dẫn đến dev team phải hỏi lại nhiều. Bạn gặp 1:1 để đưa feedback.

**Yêu cầu:**
- Bắt đầu bằng positive
- Dùng SBI cho constructive feedback
- Kết thúc bằng cách offer support
- Tone: caring, không phán xét
